## Positioning map

**Axes:** repository decision context → application/package enforcement; self-serve public → enterprise/private.

- **AI Supply Chain Trust:** public/self-serve + repository decision context.
- **Repo Trust:** local/self-serve + repository decision context.
- **Repository Trust Doctor:** local/self-serve + deep repository analysis/policy.
- **OpenSSF Scorecard:** public ecosystem standard + security-health metrics.
- **Socket:** team/enterprise + package enforcement and threat intelligence.
- **Endor Labs:** enterprise/private + broad AppSec governance and remediation.

## Strategic conclusions

1. The category is real but rapidly filling: two very recent open-source projects use nearly identical evidence-backed repository-decision language.
2. “Explainable score,” “missing evidence,” “open source,” and “MCP” are not unique claims.
3. The strongest present combination is **hosted public context + historical fix/CVE intelligence + regression leads + evidence-gated agent delivery**.
4. OpenSSF should be treated as a trusted input. Socket and Endor should be framed as complementary specialist/enterprise controls.
5. The initial wedge should be a concrete moment: **before a developer or coding agent uses an unfamiliar public repository**.
6. Urgent roadmap risks are richer live evidence coverage, first-party user proof, organization/private-repo workflows, and clearer policy mapping.
